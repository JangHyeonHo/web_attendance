## D10. 멀티테넌시 모델 — Pool(공유 스키마 + tenant_id)

- **배경**: 출결 시스템을 고객사별 유저 풀을 갖는 SaaS로 전환. 격리 수준을 정해야 함.
- **선택지**: ① 테넌트별 DB(Silo) ② 테넌트별 스키마 ③ 공유 스키마 + `tenant_id` 컬럼(Pool)
- **결정**: ③ Pool. 초기 고객 수가 적고 운영 인력이 1인인 상황에서 마이그레이션·백업·모니터링을 N벌 관리하는 비용이 격리 이득을 압도한다. 격리는 (a) 모든 쿼리의 `tenant_id` 이중 조건 (b) 매퍼 시그니처 규칙(tenantId가 항상 첫 `@Param`) (c) `UNIQUE(tenant_id, email)` 제약으로 강제.
- **기대값**: Flyway 마이그레이션 1벌, 테넌트 추가 = INSERT 1건(수 초), 전 테넌트 통계 용이.
- **리스크/비용**: 쿼리 1개의 조건 누락이 곧 데이터 유출. → 매퍼 규칙 + 실 DB 스모크(ISO 매트릭스) + 크로스 테넌트 시나리오 테스트로 방어. 대형 고객 요구시 Silo 전환 여지는 남음(테이블 구조는 동일하므로 덤프 분리로 이행 가능).

## D11. 가입 구조 — 셀프 가입 폐지, 운영자 → 고객사 관리자 → 멤버

- **배경**: B2B SaaS로 불특정 개인 가입이 없어짐(사용자 확정 사항).
- **결정**: `POST /api/v1/users`(회원가입)와 W003 화면 삭제. 운영자(SYSTEM_ADMIN)가 테넌트 등록시 고객사 관리자 계정을 동시 생성하고, 초기 비밀번호는 **응답에서 단 한 번만** 평문 반환(DB에는 BCrypt만). 고객사 관리자(TENANT_ADMIN)가 자기 테넌트의 멤버를 같은 방식으로 등록.
- **기대값**: 가입 경로가 관리 화면뿐이므로 스팸/무단 가입 원천 차단. 계정 생성 주체가 명확(감사 추적 용이).
- **리스크/비용**: 초기 비밀번호 전달이 수동(화면에서 복사→전달). Phase 3에서 초대 링크로 대체 예정(TODO 기재).

## D12. 권한 3단계 + 경로 화이트리스트 인가(RoleInterceptor)

- **배경**: 기존 is_admin 불리언으로는 운영사/고객사 관리자를 구분 못 함.
- **선택지**: ① Spring Security 도입 ② 기존 인터셉터 체계 확장
- **결정**: ② 유지. 인증(AuthInterceptor)과 인가(RoleInterceptor)를 분리하고, 인가는 **경로 접두사별 허용 role 화이트리스트**로 선언(`/system/**`→SA, `/tenant/**`→TA, `/attendance/**`→TA·MEMBER, `/admin/**`→SA). 컨트롤러 코드에 인가 로직 없음.
- 부수 결정: **SYSTEM_ADMIN은 출결 API 접근 불가**(403). 운영사 계정은 근태 당사자가 아니며, 화이트리스트에 attendance 경로를 명시적으로 등록해 "등록 누락 = 전원 통과"가 되지 않게 함.
- **기대값**: 신규 API의 인가 누락이 경로 등록 한 곳에서 리뷰됨. 화면(navigation의 role 분기)과 API 인가가 같은 role 집합을 공유.
- **리스크/비용**: 경로 규약을 지키지 않은 신규 컨트롤러는 보호 밖. → 경로 규약을 계획 문서에 명문화 + 테스트로 회귀 방지.

## D13. 결제·기업 정보 보호 — 앱 레벨 AES-256-GCM + 응답 마스킹 (사용자 확정: 암호화/마스킹 필수)

- **선택지**: ① DB TDE(투명 암호화) ② 앱 레벨 필드 암호화 ③ 둘 다
- **결정**: ② 앱 레벨(`FieldCipher`). TDE는 DB 접근자에게 평문이 보이지만, 앱 레벨은 DB 덤프·백업 유출시에도 암호문. 포맷은 `v1:{base64(iv)}:{base64(ct||tag)}` 텍스트 — 버전 접두사로 향후 키 교체/알고리즘 변경 여지 확보. 대상: 사업자등록번호, 담당자 연락처, PG 빌링키.
- 마스킹 원칙: 조회 응답은 마스킹값만(`123-**-*****`, `010-****-5678`), 필드명에 `Masked`를 붙여 오독 방지. **빌링키는 마스킹조차 없이 어떤 응답에도 미포함**(`hasBillingKey` 불리언만). DTO `toString()`에서도 민감 필드 제외(로그 유출 방지).
- 카드 원본(PAN/CVC/유효기간)은 **어떤 형태로도 저장하지 않음** — PG 빌링키 + 표시용 4자리만. PCI-DSS 적용 범위 자체를 회피.
- **기대값**: DB 유출·로그 유출·API 응답 캡처 어느 경로로도 결제 식별 정보가 평문으로 나가지 않음.
- **리스크/비용**: `APP_CRYPTO_KEY` 분실 = 해당 필드 복구 불가(운영 키 백업 절차 필요). 암호화 컬럼은 LIKE 검색 불가(검색 요구시 별도 해시 컬럼 필요). 개발 기본키로 암호화된 데이터는 운영 반입 금지(properties에 경고 주석).

## D14. 기존 데이터 이관 — V4 단일 마이그레이션, DEFAULT 테넌트 backfill

- **결정**: 신규 테이블 생성 → DEFAULT 테넌트 시드 → 컬럼 추가(NULL 허용) → backfill(유저는 DEFAULT 소속, `is_admin`→role, V2 시드 관리자는 SYSTEM_ADMIN 승격, 출결·스케쥴은 소유 유저의 테넌트를 따름) → 제약 원자 교체(단일 ALTER로 NOT NULL + FK + 신 UNIQUE 추가와 구 UNIQUE 제거) 순서. 전 구문 재실행 내성(IF NOT EXISTS/WHERE 조건)으로 부분 실패 후 이력 정리→재실행 가능.
- **검증**: 실제 V3 상태의 로컬 MariaDB에 적용 리허설. 이 과정에서 MariaDB의 FK 구문 제약(`FOREIGN KEY IF NOT EXISTS`이며 `CONSTRAINT IF NOT EXISTS`가 아님)을 발견·수정했고, 부분 실패→재실행 경로가 실제로 동작함을 확인.
- **기대값**: 기존(v2.0) 설치가 기동만으로 SaaS 스키마로 이행. 데이터 손실 없음.

## D15. 로그인 — 테넌트 코드 + 레이트 리밋

- **결정**: 로그인 식별자를 `테넌트 코드 + 이메일 + 비밀번호`로. 같은 이메일이 여러 테넌트에 존재 가능(`UNIQUE(tenant_id,email)`)하므로 코드가 필수이며, 부수적으로 코드를 모르면 계정 존재 확인 자체가 어려움. 실패 응답은 원인 무관 단일 401 메시지(계정/테넌트 존재 비노출).
- 무차별 대입 방어: 인메모리 슬라이딩 윈도우(5분) — 계정당 5회, IP당 30회 초과시 429. 성공시 계정 카운터 리셋.
- **리스크/비용**: 인메모리라 다중 인스턴스에서는 인스턴스별 카운트(TODO에 분산 대응 기재). IP는 프록시 뒤에서 부정확할 수 있음.

## D16. 랜딩 페이지(W000) — 코드가 아닌 DB 텍스트로

- **결정**: 기존 인덱스 화면을 회사/제품 소개 랜딩으로 개편. 카피 전체(히어로/기능 4종/신뢰 3종/도입 절차 3단계/CTA/푸터)를 `language_master`의 `LANDING_*` 키(3개국어, V5 시드)로 관리 — 카피 수정에 배포 불요, 언어 전환 즉시 반영. CTA는 도입 문의(mailto, 주소도 `CONTACT_EMAIL` 키)와 로그인 두 개.
- **기대값**: 비로그인 사용자의 첫 화면이 제품 소개가 되고, 영업 문의 경로가 생김. 서버 주도 화면 전개 규칙(navigate `W000`)에 그대로 편입.

## D17. 다중 에이전트 개발 프로세스 — 계획 문서 → 교차 리뷰 → 워크트리 병렬 구현 → 다관점 검증

- **결정**(사용자 지시 반영): ① 도메인 담당 에이전트 6개가 계획 문서를 병렬 작성(보안/DB이관/백엔드API/프론트/랜딩/테스트) ② 교차 리뷰 에이전트가 문서 간 모순 검출(25건 발견 — 화면코드 의미 분열, 인가 모델 모순, 암호문 포맷 3파전 등 구현 착수시 사고였을 6건 포함) ③ 정본 결정(D-A~D-F)을 문서에 일괄 반영 ④ git 워크트리 2개(백엔드/프론트)에서 병렬 구현 ⑤ 통합 후 실 DB 스모크 44건 + Playwright E2E 16스텝 + 다관점(암호화·격리·인가·프론트 계약) 적대적 리뷰.
- **기대값**: 계획 단계에서 모순을 소진해 구현 재작업 최소화. 병렬 구현으로 리드타임 단축. 리뷰 렌즈 분리로 단일 시점의 사각 제거.

## D18. 통합 후 다관점 적대적 리뷰에서 확정된 결함 8건과 수정

4개 렌즈(암호화·결제 / 테넌트 격리 / 인가·세션 / 프론트 계약)로 리뷰한 뒤 발견 전건을
별도 검증 에이전트가 반증 시도 — 8건 전부 실결함으로 확정(반증 0건), 전건 수정.

| 심각도 | 결함 | 수정 |
|--------|------|------|
| high | DEFAULT 테넌트의 TENANT_ADMIN이 같은 테넌트에 공존하는 SYSTEM_ADMIN을 비활성/강등 가능(플랫폼 관리자 잠금) | 조작 대상이 SYSTEM_ADMIN이면 404(존재 비노출), 멤버 목록에서도 제외 |
| high | 계정 비활성/강등·테넌트 정지가 기존 세션에 반영되지 않음(세션 타임아웃 20분까지 권한 잔존) | `SessionRevalidationInterceptor` 신설 — 매 요청 유저/테넌트 상태 재검증, 회수시 세션 즉시 무효화(언어만 이월), role 변경은 스냅샷 즉시 갱신 |
| medium | 위와 동일 근원(격리 렌즈에서 독립 발견) | 상동 |
| medium | 리로드 후 프론트 lang 상태와 서버 세션 언어 불일치(임베드 화면 언어 혼합) | `NavigateResponse`에 서버 확정 `lang` 동봉, 프론트는 그 값으로 동기화 |
| medium | navigate 동시 호출시 늦은 응답이 최종 화면을 덮는 레이스 | 요청 시퀀스 토큰으로 최신 응답만 반영 |
| low | 개발 기본 암호키(저장소 커밋 값)로 조용히 암호화될 수 있음 | dev 키 사용시 기동 WARN 로그(prod 프로파일은 기존대로 키 미설정시 기동 실패). 키 버전 식별자(kid) 도입은 후속 과제 |
| low | 첫 navigation 실패시 '...' 화면에서 복구 불가 | navigate 오류 상태 도입 — 부팅 실패는 재시도 화면, 이후 실패는 배너 |
| low | 결제/기업 정보 편집 '취소' 경로에서 빌링키·사업자번호 평문이 state 잔존 | 취소 핸들러에서도 민감 입력 즉시 클리어(제출 경로와 동일 정책) |

## D19. 테넌트 서브도메인 병행 방식 (코드 입력과 혼용)

- **배경**: 로그인 시 테넌트 코드 입력을 서브도메인(`acme.webatt.example`) 방식으로 대체하고 싶으나,
  서버/인프라(와일드카드 DNS·TLS) 구축 일정이 불확실.
- **선택지**: ① 코드 방식 유지 ② 서브도메인 전면 전환 ③ 병행(호스트 우선, 없으면 코드)
- **결정**: ③ 병행. `app.tenant.base-domain` 미설정(기본)이면 기존 코드 방식만 동작하고,
  설정하면 서브도메인이 켜진다 — **인프라 준비와 코드 배포를 분리**하는 무중단 전환 경로.
- 규칙: **호스트가 테넌트를 주장하면 호스트가 이긴다.** 바디 코드는 생략 가능하되 다르면 400(모호성 명시 거부).
  미등록 서브도메인은 통일 401(존재 비노출), 예약어(www/admin/api/app/mail)는 해석 제외 + 테넌트 코드 등록 차단
  (서브도메인을 켜기 전에 선점되는 사고 방지 — 병행 시작 시점부터 적용).
- 보안: 쿠키는 host-only(서블릿 기본) + **세션-호스트 일치를 서버가 강제**(`SessionRevalidationInterceptor`) —
  다른 테넌트 서브도메인으로 이식된 세션 쿠키는 즉시 무효화. 이중 방어.
- UX: 테넌트 서브도메인의 비로그인 초기 화면은 W001(로그인, 회사명 표시 — navigation `hostTenantName`),
  랜딩(W000)은 루트 도메인 전용. 프론트는 호스트 해석 여부로 코드 입력란을 자동 표시/숨김.
- **기대값**: 서버 구축 전에도 기능이 완성·검증돼 있고, DNS/TLS가 준비되는 순간 환경변수 하나로 활성화.
  고객별 전용 URL 배포 후에도 코드 방식이 깨지지 않아 안내 전환이 유연.
- **리스크/비용**: 리버스 프록시의 Host 전달 설정 누락 시 전 요청이 루트 취급(기능 미동작으로 나타나 안전한 실패).
  서브도메인은 전 플랜 기본 제공 방침(과금은 커스텀 도메인에서 — Phase 4 참고 결정).
