## D1. Spring Boot 4.1.0 / Java 21

- **배경**: Boot 2.5.2(2021, EOL) + Java 8. 보안 패치가 끊긴 상태.
- **선택지**: ① Boot 3.5(안정 우선) ② Boot 4.1(최신 GA)
- **결정**: 1차 패치에서 3.5로 올려 안정성을 확인한 뒤, 2차(v2.0.0)에서 4.1.0으로. MyBatis 스타터 4.0.1과 springdoc 3.0.3이 Boot 4 지원을 시작했음을 확인하고 진행.
- **기대값**: 지원 기간 확보(보안 패치 수령), Java 21 문법(record, 패턴매칭, 화살표 switch) 사용, 최신 생태계 호환.
- **리스크/비용**: Boot 4는 자동설정이 모듈로 분리되어 의존성 추가가 필요한 경우가 있음(실제로 Flyway가 `spring-boot-starter-flyway` 분리로 초기 미동작 → 수정). 마이너 버전업시 릴리즈 노트 확인 필요.

## D2. Oracle → MariaDB 전환 + Flyway

- **배경**: 기존 Oracle RDS는 폐기됨. 스키마 관리는 자바 코드로 DDL을 생성·비교하는 자작 시스템(`AdminSettingLogic` 717줄).
- **선택지**: ① Oracle 재구축 ② MariaDB + 자작 시스템 이식 ③ MariaDB + Flyway
- **결정**: ③. 데이터가 없으니 스키마를 처음부터 재설계(스네이크케이스, AUTO_INCREMENT PK, 적절한 타입), 버전 관리는 표준 도구에 위임.
- **기대값**: 무료 라이선스·컨테이너 친화(로컬 `docker compose up`으로 개발 환경 완성), 스키마 이력의 git 추적(`db/migration/V*.sql`), 어떤 DB 인스턴스든 기동만 하면 동일 스키마 보장, 자작 DDL 코드 717줄 유지보수 소멸.
- **리스크/비용**: Oracle 전용 SQL(`CONNECT BY`, `NVL`, `DECODE`) 전면 재작성 필요했음(완료). 적용된 마이그레이션 파일은 수정 불가라는 규칙 학습 필요.

## D3. 단일 `/api` → 리소스별 REST + record DTO (Map 남용 제거)

- **배경**: 모든 요청이 `Map<String,Object>`로 들어와 `win_id`/`action`으로 분기. 타입 안전성 없음(형변환 유틸 `objectTo*` 필요), API 형태가 코드 밖에서 보이지 않음, Swagger를 붙여도 "Map 하나"로만 표시.
- **선택지**: ① Map 유지 + Swagger 설명만 수기 작성 ② 리소스별 엔드포인트 + 타입 DTO
- **결정**: ②. 요청/응답/도메인을 record로 정의, Bean Validation으로 입력 검증, 오류는 HTTP 상태코드 + 구조화된 `ErrorResponse`.
- **기대값**: 컴파일 타임 타입 검증, Swagger 자동 문서화(스키마·예시 포함), 필드 단위 검증 에러 응답, 프론트 개발시 계약이 명확.
- **리스크/비용**: **구 API와 호환 불가** — 프론트 전면 수정 필요(어차피 재작성 대상이라 수용). 화면 전개 컨셉이 사라지는 부작용 → D7로 복원.

## D4. MyBatis 유지 + XML → 어노테이션 매퍼

- **배경**: "MyBatis 폐기?"로 보일 수 있는 부분. 폐기 아님.
- **선택지**: ① XML 매퍼 유지 ② 어노테이션 매퍼 ③ JPA 전환
- **결정**: ②. JPA 전환은 학습 비용과 SQL 제어력 상실 대비 이득이 없어 배제. 재설계 후 SQL이 전부 단순 CRUD가 되었고(유일한 복잡 쿼리였던 달력 SQL은 자바로 이동), XML은 네임스페이스 매칭 간접층만 남기 때문.
- **기대값**: SQL이 인터페이스 옆에 위치(탐색 비용 감소), 파일 수 감소, v1에서 XML이 공개 폴더에 노출됐던 유형의 사고 원천 차단. record 매핑은 `arg-name-based-constructor-auto-mapping`으로 해결.
- **리스크/비용**: 복잡한 동적 SQL/조인 resultMap이 필요해지면 어노테이션이 불리 → **그 매퍼만 XML로 복귀 가능**(공존 지원)하므로 낮음.

## D5. 비밀번호 SHA-512(무솔트) → BCrypt

- **배경**: v1은 솔트 없는 SHA-512 단방향 해시. 레인보우 테이블에 취약. 게다가 해시를 로그로 출력.
- **결정**: DB 폐기로 기존 해시와의 호환 제약이 사라진 지금이 전환 적기. `spring-security-crypto`만 도입(전체 Spring Security는 세션 인증 구조에 과함).
- **기대값**: 솔트 자동 내장 + 연산 비용 조절 가능한 표준 해시. 로그인 실패 메시지 단일화(이메일 존재 여부 비노출), 로그인시 세션 재발급(세션 고정 공격 방지)도 함께 적용.
- **리스크/비용**: 없음(기존 사용자 데이터가 없으므로).

## D6. 도메인(기능별) 패키지 구조

- **배경**: 레이어별 구조(controller/service/dao/dto)는 기능 하나가 4개 패키지에 분산, `other` 같은 잡동사니 패키지 발생.
- **결정**: package-by-feature(`auth`/`user`/`attendance`/`language`/`navigation`/`common`/`config`).
- **기대값**: 기능 단위 응집(navigation 추가가 폴더 하나로 완결된 것이 실증), 기능간 의존이 import로 가시화, 모듈 분리 용이.
- **리스크/비용**: 레이어드에 익숙하면 초기 위화감. 규모가 커지면 기능 내부에 하위 패키지 필요.

## D7. 서버 주도 화면 전개(Navigation) 복원

- **배경**: D3 재설계로 v1의 핵심 컨셉(서버가 표시 화면을 결정, 화면 명은 W-코드로 은닉, URL 미사용)이 사라짐. 이 컨셉은 의도된 설계 목표였음.
- **선택지**: ① URL 라우팅(일반적 SPA 관례) ② 서버 주도 + 화면 코드(v1 계승)
- **결정**: ②를 타입 있는 형태로 복원(`POST /api/v1/navigation` + `Screen` enum 레지스트리 + 전환 사유 `reason`). 결정 규칙은 순수 함수로 분리해 단위 테스트.
- **기대값**: 화면 흐름(인가 리다이렉트, 권한 게이팅)이 서버 한 곳에 집중 — 배포 없이 흐름 변경 가능, 클라이언트가 얇아짐, 화면 명 은닉 유지. v1에서 테스트용으로 풀려 있던 관리자 화면 보호도 이번에 활성화.
- **리스크/비용**: 화면 전환마다 왕복 1회 추가, 서버가 UI 흐름과 결합(화면 추가시 Screen enum 수정 필요), 브라우저 히스토리/딥링크 미지원은 이 설계의 의도된 트레이드오프.

## D8. 3개 언어(한/영/일) 대응 — MessageSource + springdoc i18n

- **배경**: v1은 DB 언어 마스터(화면 텍스트)만 한/영 지원, 일본어는 한국어 폴백. 서버가 만드는 메시지(에러/검증/출결 상태·확인 메시지)는 전부 한국어 하드코딩. Swagger 도입 후 문서도 한국어 고정.
- **선택지**: ① DB 언어 마스터에 서버 메시지도 등록 ② 표준 MessageSource 프로퍼티 번들
- **결정**: ②. 서버 메시지는 코드와 함께 배포되는 정적 자원이므로 번들(`messages_ko/en/ja.properties`)이 적합. DB 언어 마스터는 "화면 텍스트"(운영 중 등록/수정) 용도로 역할 분리 유지(JPN 등록 가능하도록 확장).
  - 로케일 결정: **세션 언어(navigation의 lang) → Accept-Language → 한국어** 순. 커스텀 LocaleResolver로 일원화.
  - Bean Validation 메시지는 `{메시지키}` 방식으로 같은 번들 사용.
  - Swagger는 springdoc의 MessageSource 해석 기능 활용: @Tag/@Operation/@ApiResponse에 메시지 키를 쓰고 `springdoc.cache.disabled=true`로 요청 언어별 재해석.
- **기대값**: 같은 API가 요청 언어에 따라 에러 메시지·상태 라벨·확인 메시지·Swagger 문서까지 번역되어 응답. 번역 누락은 키 동기화 테스트(`MessagesTest.bundlesHaveSameKeys`)가 빌드에서 검출.
- **리스크/비용**:
  - `springdoc.cache.disabled=true`는 문서 요청마다 스펙을 재생성(운영에서 문서 노출 안 하면 무관).
  - OpenAPI `Info`는 springdoc i18n 대상이 아니라서 3개 언어 병기로 처리.
  - 로그인시 세션 재발급 과정에서 언어 설정이 소실되는 버그를 실기동 검증에서 발견, 언어 승계 처리로 수정(로그아웃도 동일).

### D8-1. 하드코딩 메시지 전량 제거 (후속 작업, 완료)

- **배경**: D8 이후에도 소스에 하드코딩 메시지가 남아 있었다 — 예외 발생부의 한국어 문자열(서비스가 즉시 번역해서 던지는 방식), `@Schema`/`@Parameter` 설명 약 95곳.
- **결정**:
  - **예외는 메시지 키 지연 해석 패턴으로 통일**: `ApiException`이 문자열 대신 `messageKey + args`를 보관하고, `GlobalExceptionHandler`가 응답 시점에 요청 로케일로 해석. 서비스/인터셉터에서 번역 책임이 사라져 Messages 의존이 응답 본문 생성부(라벨/성공 메시지)로만 축소됨.
  - `@Schema`/`@Parameter` 설명 전부(약 75개 고유 키)를 메시지 키로 전환.
  - springdoc의 스키마 자동 해석(`resolve-schema-properties`)은 **JVM 기본 로케일에 고정되는 한계를 실측으로 확인** → 사용하지 않고, 요청 로케일로 해석하는 커스텀 `OpenApiCustomizer`를 구현.
- **기대값**: 소스 코드에 사용자 노출 문자열이 존재하지 않음(메시지는 전부 번들에서 관리). Swagger의 모델/필드 설명까지 요청 언어를 따름. 번역 추가/수정이 properties 파일 편집만으로 완결.
- **잔여**: `@Schema`의 `example` 값(예: "홍길동")은 메시지가 아닌 예시 데이터라 의도적으로 유지.

## D9. 검증 전략 — 단위 테스트 + 로컬 실기동 스모크

- **배경**: v1은 테스트 0건(컨텍스트 로드 테스트는 잘못된 패키지 + DB 필요로 사실상 동작 불가).
- **결정**: DB 없이 도는 단위 테스트(상태머신 규칙, 월별 페어링, 화면 전개 결정, 메시지 번들 — 35건)를 CI 기준으로 삼고, 통합 검증은 로컬 MariaDB 실기동 + 전 API 스모크(curl)로 수행. DB 필요한 컨텍스트 테스트는 `@Disabled` 유지.
- **기대값**: 핵심 비즈니스 규칙의 회귀 방지, `mvn test`가 어디서든 통과.
- **리스크/비용**: 매퍼 SQL은 단위 테스트 범위 밖(스모크로 보완). 후속 과제로 Testcontainers 검토 가능.

## 남은 리스크 / 후속 과제

- `@Schema`(DTO 필드 설명)의 3개 언어화
- 운영 프로파일(Swagger/devtools 비활성, 세션 저장소) 분리
- 매퍼 SQL 통합 테스트(Testcontainers)
- 프론트 재작성(navigation 기반) — 점검 보고서 참조
- git 이력에 남은 구 RDS 접속 정보 → 해당 계정 폐기 필수(이력 정리는 별도 판단)
