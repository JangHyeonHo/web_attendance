# 13. 테스트 서버 운영 정책

> [인덱스](README.md)

테스트 공개 기간의 운영 원칙. 실판매가 없는 단계이므로 "정식 서비스로 오해받지 않는 것"과
"테스트 데이터라는 이유로 소홀히 다루지 않는 것" 사이의 선을 정해 둔다.

## 표기 원칙 (정식 서비스로 오해받지 않기)

- 랜딩·화면에 **테스트 운영 중**임을 명시한다(문구·배치는 랜딩 리메이크 요구정의에서 확정 —
  판매용 페이지와 로그인의 분리도 그때 함께).
- 요금·구매를 연상시키는 표기를 두지 않는다. 청구서 화면은 기능 확인용임을 전제로 안내.
- 문의 창구(이메일 등) 1개를 랜딩에 명시한다 — 테스트 참여자의 피드백 수집 경로를 겸한다.

## 데이터 취급 (테스트여도 실명 데이터다)

- 테스트 참여 회사의 출퇴근 기록·좌표는 **실제 개인정보다** — "테스트 데이터니까"로 취급 수준을
  낮추지 않는다(백업·암호화 키 규칙은 [10](10-backup-restore.md) 그대로 적용).
- 참여자에게 사전 고지할 것: 테스트 기간 종료 시 데이터가 삭제될 수 있다는 것,
  위치 정보(좌표)가 기록된다는 것.
- **데이터 초기화는 예고 후에만** 한다(참여 회사에 날짜 공지 → 백업 → 초기화).
  초기화 절차: 서비스 정지 → DB 드롭·재생성([04](04-software-install.md)) → 재기동(Flyway 재시드)
  → 시드 관리자 비밀번호 변경([08](08-initial-setup.md)) 재실행.

## 지원·장애 대응 수준 (기대치 설정)

- 가용성 목표를 약속하지 않는다 — 단일 서버라 패치 재부팅·장애로 수 분~수 시간 중단이
  있을 수 있음을 랜딩 고지에 포함.
- 장애 인지 시: 상태를 문의 창구로 공지 → 복구([09](09-deploy-update.md)/[10](10-backup-restore.md)) → 원인을 patch-notes에 기록.
- 피드백 처리: 접수 → 이슈화(GitHub) → 반영 시 참여자에게 회신. 반영 여부 판단 기준은
  "정식 서비스에도 필요한가".

## 테스트 종료 시

- [ ] 참여 회사에 종료·데이터 처리 방침 공지
- [ ] 최종 백업 1부 보관(보존 기한을 정해서) 후 서버 데이터 삭제
