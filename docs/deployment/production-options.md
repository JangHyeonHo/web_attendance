# 운영 서버 — 구성 선택지 개요 (2026-07)

> 상태: **의사결정 대기.** 운영 이전은 당분간 없고, 개발(테스트) 서버는 OCI 단일 서버로 확정
> (`dev-server-oci.md`). 이 문서는 나중에 운영을 올릴 때 "무엇 중에서 고르는 것이고,
> 우리 규모에는 무엇이 맞는가"를 판단할 수 있게 선택지와 근거만 정리해 둔다.

## 판단에 영향을 주는 우리 시스템의 사실

1. **세션이 서버 인메모리다** — 앱 인스턴스를 2대 이상으로 늘리려면 세션 공유 저장소
   (Spring Session JDBC/Redis) 도입이 선행돼야 한다(application.properties에 명시된 제약).
   즉 "일단 1대"는 코드 그대로 가능, "여러 대"는 코드 변경이 먼저다.
2. **JVM 상시 구동형**이다 — 요청 올 때만 깨어나는 서버리스(콜드 스타트)와 궁합이 나쁘다.
3. **DB는 MariaDB 단일**이고 Flyway가 스키마를 관리한다 — DB를 어디에 두든 이식은 쉽다.
4. 고객이 한·일 양국이다 — 리전은 주력 시장 쪽(도쿄/서울), 나머지는 지연 수십 ms 감수.
5. 테넌트 서브도메인(와일드카드 DNS/TLS)을 쓴다 — LB/CDN을 앞에 둘 경우 와일드카드 인증서
   지원 여부를 확인해야 한다.

## 선택지

### ① 단일 IaaS 스케일업 (개발 서버 구성 그대로 크게)

- 개발 서버와 동일 구성(nginx+jar+MariaDB 한 대)을 사양만 올려 운영으로.
- **장점**: 지금 문서·경험 그대로, 비용 최소, 운영 대상 1개.
- **단점**: 그 한 대가 곧 서비스 전체 — 디스크 장애=데이터 사고, 패치 재부팅=서비스 중단.
  백업·복구가 전부 수동 책임.
- **적합**: 무료 베타~극초기(수 개 테넌트, 중단이 사과로 끝나는 시기).

### ② IaaS 앱 서버 + 매니지드 DB ★권장 경로★

- 앱(nginx+jar)은 IaaS 1~2대, DB만 매니지드(OCI MySQL/HeatWave, AWS RDS MariaDB 등)로 분리.
- **장점**: 운영 사고의 대부분(백업, 패치, 디스크, 장애 복구)이 DB에서 나는데 그걸 통째로
  위임한다. 자동 백업·시점 복원(PITR)·모니터링이 기본 제공. 앱 서버는 상태가 없어져
  (세션 공유 도입 후) 2대+LB로 무중단 배포·장애 내성 확보가 가능해진다.
- **단점**: 비용 증가(매니지드 DB가 인스턴스보다 비쌈), 구성 요소 2~3개로 증가.
- **적합**: 유료 고객이 생기는 시점부터. **우리 규모(초기 수십 테넌트, 사내 근태라 트래픽
  예측 가능·낮음)에는 이 단계가 종착점에 가깝다** — 이 이상은 과잉이 되기 쉽다.

### ③ 컨테이너 서버리스 (Cloud Run / App Runner 류)

- 컨테이너 이미지를 주면 스케일링·LB를 플랫폼이 해 주는 방식.
- **장점**: 서버 관리 소멸, 트래픽 비례 과금.
- **단점**: 콜드 스타트가 JVM과 상성이 나쁘고(첫 요청 수 초), 상시 1대 유지를 걸면 비용 이점이
  줄어든다. 인메모리 세션 불가(세션 공유 필수). 와일드카드 커스텀 도메인 제약도 플랫폼별 확인 필요.
- **적합**: 트래픽 기복이 큰 B2C. 근태처럼 출퇴근 시각에만 몰리는 예측 가능한 부하에는 이점이 작다.

### ④ 쿠버네티스 (K8s)

- **우리 규모에서는 명확히 과잉.** 학습·운영 비용이 서비스 개발을 잠식한다. 앱이 수 개 서비스로
  쪼개지고 팀이 커진 뒤의 이야기.

### ⑤ 소형 PaaS (Fly.io / Railway 류)

- git push로 배포되는 간편함이 장점이나, MariaDB·와일드카드 도메인·리전 선택의 자유도가 낮고
  플랫폼 락인이 생긴다. 이미 OCI 단일 서버 구축 경험을 갖게 되는 우리에게는 이점이 애매하다.

## 권장 로드맵

```
[지금] 개발/테스트: OCI 단일 서버 (①과 동일 구성)
   │  테스트 운영으로 부하·운영 감각 파악
   ▼
[운영 개시] ②-축소형: 앱 IaaS 1대 + 매니지드 DB
   │  코드 변경 없음(1대이므로 세션 그대로). DB 리스크만 먼저 위임
   ▼
[유료 고객 증가] ②-완성형: Spring Session 도입 → 앱 2대 + LB (무중단 배포·장애 내성)
```

이렇게 가는 이유: 각 단계가 이전 단계의 구성·문서를 버리지 않고 한 조각씩만 추가하며,
코드 변경(세션 공유)이 필요한 시점을 가장 뒤로 미룰 수 있다. 반대로 ③~⑤로 시작하면
초기부터 코드·플랫폼 제약을 함께 떠안는다.
