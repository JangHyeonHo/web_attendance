# 03. EC2 인스턴스(서버) 만들기

> 이전: [02. AWS 계정 준비](02-aws-iam.md) · 다음: [04. 소프트웨어 설치](04-software-install.md) / [인덱스](README.md)

이 문서가 끝나면: 고정 IP가 붙은 우분투 서버 1대가 생기고, 내 PC에서 SSH로 들어갈 수 있다.
작업은 전부 **IAM 사용자로 로그인한 콘솔**에서 한다(루트 아님 — [02](02-aws-iam.md)).

## 3-1. 리전 선택

콘솔 **우상단의 리전 선택**에서 **아시아 태평양(도쿄) ap-northeast-1** 선택.
(서울 ap-northeast-2도 절차는 완전히 동일 — 주 사용 위치에 가까운 쪽으로.
이후 화면은 전부 이 리전 안에서의 작업이다 — 리전이 다르면 만든 것이 안 보이니 주의.)

## 3-2. 키 페어 만들기 (서버 열쇠)

서버 접속은 비밀번호가 아니라 **키 파일**로 한다:

1. 콘솔 검색창 "EC2" → 왼쪽 메뉴 **네트워크 및 보안 → 키 페어(Key Pairs)** → **키 페어 생성**
2. 이름: `mt-dev-key` / 유형: **ED25519** / 형식: **.pem**
3. 생성 즉시 `mt-dev-key.pem` 파일이 다운로드된다 —
   ★이 파일이 곧 서버 열쇠. 다시 받을 수 없으니 안전한 폴더에 보관, 유출 금지★

## 3-3. 인스턴스 시작

EC2 → **인스턴스(Instances)** → **인스턴스 시작(Launch instances)**.
기본값에서 **바꿔야 할 것만** 나열한다:

| 항목 | 할 일 |
|---|---|
| 이름 | 예) `mt-dev-01` |
| **AMI(이미지)** | **Ubuntu Server 24.04 LTS** 선택 → 아키텍처를 **64비트(Arm)** 로 변경 |
| **인스턴스 유형** | **t4g.small**(2 vCPU/2GB — 권장 시작점) 또는 t4g.medium(4GB — 여유) |
| **키 페어** | 3-2에서 만든 `mt-dev-key` 선택 |
| 네트워크 설정 | "보안 그룹 생성" 상태에서 체크 3개: **SSH 트래픽 허용**(소스: **내 IP**), **인터넷에서 HTTPS 트래픽 허용**, **인터넷에서 HTTP 트래픽 허용** |
| 스토리지 | **30 GiB, gp3** |

우측 요약에서 **인스턴스 시작** → 목록에서 상태가 "실행 중(Running)"이 되면 완료.

- 인스턴스 유형은 나중에 **중지 → 유형 변경 → 시작**으로 몇 분 만에 키울 수 있다 —
  작게 시작해도 된다.
- 참고 월 비용(도쿄, 대략): t4g.small 컴퓨팅 ~$16 + 디스크 30GB ~$3 + 공인 IPv4 ~$4 =
  **월 $22~25(약 3만 원 초반/¥3,500 전후)**. t4g.medium이면 컴퓨팅이 2배.

## 3-4. 고정 IP(Elastic IP) 붙이기

기본 공인 IP는 인스턴스를 중지/시작할 때마다 바뀐다. DNS에 등록할 것이므로 고정한다:

1. EC2 왼쪽 메뉴 **네트워크 및 보안 → 탄력적 IP(Elastic IPs)** → **탄력적 IP 주소 할당** → 할당
2. 생성된 주소 선택 → **작업 → 탄력적 IP 주소 연결** → 인스턴스: `mt-dev-01` 선택 → 연결
3. 이 주소를 메모 — **이후 문서의 `<공인 IP>`는 전부 이 값**
   (IPv4 요금 ~$4/월은 고정 여부와 무관하게 공인 IPv4를 쓰는 순간 붙는 요금이라,
   Elastic IP로 바꿔도 추가 비용은 없다)

## 3-5. 첫 SSH 접속

**내 PC에서**, `.pem` 파일이 있는 폴더에서:

```
ssh -i mt-dev-key.pem ubuntu@<공인 IP>
```

- 처음 접속 시 "fingerprint ... yes/no" → `yes`
- Mac/리눅스에서 "Permissions ... too open" 오류가 나면: `chmod 400 mt-dev-key.pem` 후 재시도
- Windows에서 "bad permissions" 오류가 나면: 파일 우클릭 → 속성 → 보안 → 고급 →
  상속 사용 안 함 → 본인 계정만 남기기
- 프롬프트가 `ubuntu@ip-...`로 바뀌면 서버 안이다. 첫 갱신:

```bash
sudo apt update && sudo apt -y upgrade
```

## 3-6. 방화벽에 대해 알아둘 것

- AWS의 방화벽은 **보안 그룹** 하나다(3-3에서 22/80/443을 이미 열었다).
  OS 안에 별도 방화벽은 기본 비활성이라 추가 작업이 없다.
- **3306(DB)은 열지 않는다** — DB는 서버 안(127.0.0.1)에서만 접속한다.
  이후 어떤 이유로도 보안 그룹에 3306 인바운드를 추가하지 말 것.
- 집/사무실 IP가 바뀌어 SSH가 안 되면: EC2 → 보안 그룹 → 인바운드 규칙 편집에서
  SSH 규칙의 소스를 현재 "내 IP"로 갱신한다.

## 완료 확인

- [ ] 인스턴스 "실행 중", 유형 t4g.small(또는 medium), Ubuntu 24.04 Arm
- [ ] Elastic IP가 인스턴스에 연결됨
- [ ] `ssh -i mt-dev-key.pem ubuntu@<공인 IP>` 접속 성공
- [ ] 보안 그룹 인바운드가 22(내 IP)/80/443 세 개뿐
