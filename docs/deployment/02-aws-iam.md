# 02. AWS 계정 준비 — 계정 설계와 IAM

> 이전: [01. 전체 구성](01-architecture.md) · 다음: [03. 네트워크(VPC)](03-aws-network.md) / [인덱스](README.md)
> 모르는 용어는 [용어집](glossary.md).

전제: **AWS 루트 계정(가입 시 이메일로 로그인하는 계정)은 이미 있다.**

## 2-0. 이 계정의 사용 설계 (먼저 그림부터)

이 AWS 계정에서 "누가, 무엇을, 어떤 권한으로" 하는지를 먼저 정한다.
사용자 계정은 이 표에서 도출된 것만 만든다 — 표에 없는 계정은 만들지 않는다.

| 주체 | 하는 일 | 권한 | 시점 |
|---|---|---|---|
| **루트** | 루트 전용 스위치 조작(결제 접근 허용 등)뿐 | (전능 — 그래서 봉인) | 이번 1회, 이후 비상시만 |
| **admin-<이름>** (IAM 사용자) | 구축·설정 변경 전반(03~ 문서의 콘솔 작업 전부) | AdministratorAccess | 지금 생성 |
| **op-<이름>** (IAM 사용자) | 테스트 날 서버 켜고 끄기만([15](15-on-demand.md)) | EC2 시작/정지 한정(아래 2-3 정책) | **온디맨드 운용을 타인에게 맡기거나, 본인이 폰에서 안전하게 켜고 끌 때** 생성 |
| 향후 협업자 | 개발·운영 참여 | 그때 업무 범위로 최소 설계 | 발급 규칙(2-3)에 따름 |

설계 원칙:
- **1인 1계정** — 계정 공유 금지(누가 뭘 했는지 추적 불가능해짐).
  참고로 AWS는 관리 작업 이력(CloudTrail 이벤트 90일)을 기본으로 남기고 있어,
  계정만 분리돼 있으면 "언제 누가 껐다 켰나"는 콘솔에서 추적된다
- **권한은 용도에서 도출** — "일단 관리자" 금지. 위 표처럼 하는 일을 먼저 적고 권한을 붙인다
- **일상 작업에 admin을 쓰지 않게 되는 것이 최종 상태** — 구축이 끝나면 일상 조작은
  켜고 끄기(op 계정)뿐이므로, admin은 설정 변경 때만 꺼내는 두 번째 열쇠가 된다
- IAM Identity Center(SSO)는 채택하지 않는다 — 다인원·다계정 조직용이라 현 단계에선
  관리 대상만 늘린다. 협업자가 3인을 넘거나 운영 계정을 분리하는 시점에 재검토

> **운영자 결정(2026-07): MFA와 예산 알림은 당분간 보류.**
> 단독 사용 단계라는 판단 — 대신 IAM 비밀번호는 다른 곳에서 안 쓰는 강한 것으로 유지한다.
> **타 사용자에게 계정을 발급하는 시점부터 MFA 필수**(2-3의 발급 절차에 포함).

## 2-1. 루트로 로그인해서 한 번에 끝낼 일

**루트가 아니면 못 하는 작업은 이 절이 전부다** — 한 번의 로그인으로 끝내고 봉인한다.
https://console.aws.amazon.com 에 루트(이메일)로 로그인 후:

### (a) 계정 ID 메모

우상단 계정 이름 클릭 → **계정 ID(12자리 숫자)** 메모. IAM 로그인 URL에 계속 쓴다.

### (b) 결제 콘솔 접근 허용 ★루트 전용 스위치★

이걸 안 켜면 AdministratorAccess로도 결제·프리 티어 페이지가 안 보인다(기본값 차단):

1. 우상단 계정 이름 → **계정(Account)**
2. **"결제 정보에 대한 IAM 사용자 및 역할 액세스"** → 편집 → **활성화** → 업데이트

### (c) 관리자 IAM 사용자(admin-<이름>) 생성

1. 콘솔 검색창 "IAM" → 왼쪽 **사용자(Users)** → **사용자 생성**
2. 사용자 이름 `admin-<이름>` →
   **"AWS Management Console에 대한 사용자 액세스 권한 제공" 체크** →
   "IAM 사용자를 생성하고 싶음" → 비밀번호 직접 설정
   ("다음 로그인 시 새 비밀번호 생성"은 본인용이므로 체크 해제 가능)
3. **직접 정책 연결** → `AdministratorAccess` **접미사 없는 그 이름 하나만** 체크
   (`-Amplify`, `-AWSElasticBeanstalk` 등 변종은 특정 서비스 전용 — 전부 무시) → 생성
4. **콘솔 로그인 URL** 북마크: `https://<계정ID>.signin.aws.amazon.com/console`

### (d) (보류 중) 루트 MFA

재개하는 날: 우상단 계정 이름 → 보안 자격 증명 → MFA 디바이스 할당 → 인증 앱 QR →
6자리 2회. ★루트 MFA 분실은 복구가 매우 번거로움 — 휴대폰 교체 시 인증 앱 이전 필수★

**여기까지 끝나면 루트 로그아웃. 이후 문서 전부(03~)는 IAM 사용자로만 작업한다.**

## 2-2. admin 사용자 첫 로그인과 기본 위생

북마크한 URL에서 계정 ID / `admin-<이름>` / 비밀번호로 로그인한 뒤:

1. **비밀번호 정책 강제**(사용자가 늘기 전에 미리): IAM → 왼쪽 **계정 설정(Account settings)** →
   비밀번호 정책 **편집** → 최소 길이 14+, 대/소/숫자/기호 요구, 재사용 제한 체크 → 저장.
   나중에 만들 모든 사용자에게 자동 적용된다
2. **액세스 키는 만들지 않는다** — 이 구축은 콘솔+SSH만으로 끝난다. 쓰지 않는 키는
   유출 통로일 뿐이며, CLI가 필요해지는 시점에 그 용도 한정으로 만든다
3. (보류 중) 본인 MFA: 재개 시 IAM → 사용자 → 본인 → 보안 자격 증명 탭에서 등록

## 2-3. 앞으로의 계정 발급 규칙 (타인에게 줄 때 이대로)

1. **발급 전 확인**: 그 사람이 실제로 하는 일이 무엇인가 → 2-0 표에 행을 추가
2. **MFA 필수**: 생성 시 "다음 로그인 시 비밀번호 변경" 체크 + 첫 로그인 자리에서 MFA 등록까지
   확인하고 넘긴다
3. **권한은 정책으로 좁혀서**. 예: 서버 켜고 끄기만 하는 운용 계정(op-<이름>)은
   AdministratorAccess가 아니라 아래 정책을 "인라인 정책"으로 붙인다
   (IAM → 사용자 → 권한 추가 → 인라인 정책 → JSON — `<리전>`·`<인스턴스ID>`만 치환):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow",
      "Action": ["ec2:DescribeInstances", "ec2:DescribeInstanceStatus"],
      "Resource": "*" },
    { "Effect": "Allow",
      "Action": ["ec2:StartInstances", "ec2:StopInstances"],
      "Resource": "arn:aws:ec2:<리전>:<계정ID>:instance/<인스턴스ID>" }
  ]
}
```

   이 계정은 그 서버 1대를 켜고 끄는 것 외에 아무것도 못 한다 — 실수·유출의 피해 범위가
   그만큼으로 끝난다. (본인이 모바일 앱으로 켜고 끌 때도 admin 대신 이 계정을 쓰면 안전)
4. **회수**: 역할이 끝난 계정은 비활성화(콘솔 액세스 제거)→ 30일 뒤 삭제

## 2-4. (보류 중) 요금 알림

재개 시: 검색창 "Budgets" → 예산 생성 → 월별 비용 예산(서버 예상 월액보다 약간 위, 예: $10)
→ 알림 이메일 → 생성.
