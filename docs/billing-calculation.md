# 결제(과금) 계산 방식 — 현행 동작 + 케이스별 정리 + 개선 제안

- 대상: `com.attendance.pro.billing` + `tenant_billing`/`tenant_seat_usage`/`invoice` (Flyway V22·V24)
- 이 문서의 구성
  1. **현행(As-Is)** 계산 방식 — 지금 코드가 실제로 하는 것
  2. **케이스별 결제 상황** — 가입·증원·감원·무료↔유료 전환·단가 변경 등에서 현행이 어떻게 동작하는가
  3. **문제의식** — 왜 현행(월 고정) 방식이 메리트가 약한가
  4. **개선 제안** — 일할계산(seat-day) 모델 + 전환 비대칭 처리(제안, **미구현**)
  5. **미결정 사항** — 도입 전 정해야 할 정책
- ⚠️ 표기 규칙: **[현행]** = 지금 배포된 동작 / **[제안]** = 아직 구현 안 됨(정책 확정 후 별도 작업)

## 1. 현행(As-Is) 요약

| 항목 | 값 / 방식 |
|------|-----------|
| 과금 기준 | **활성 멤버 수(seat)** — `ACTIVE`·미삭제·`SYSTEM_ADMIN` 제외 |
| 집계 주기 | **달력 월**(`YYYY-MM`), 월중 **최대치(high-water mark)** |
| 인당 단가 | 기본 **2,000원/월** (VAT 별도) — 운영사가 회사별 조정 |
| 무료 좌석 | 기본 **5좌석** — 이 수까지 과금 제외 |
| 일할계산 | **없음** (그 달 하루라도 활성이면 피크 좌석으로 한 달 전액) |
| 확정(마감) | 운영사 **수동** 마감 시 스냅샷, 자동 스케줄러 없음 |
| 결제일·자동결제·알림 | **없음** — 청구서 계산·표시까지만 (PG 미연동) |

**[현행] 과금식** (`BillingService.compute`)
```
과금인원 = max(0, 월중최대활성 − 무료좌석)
공급가   = 과금인원 × 인당단가
부가세   = round(공급가 × 10%)
합계     = 공급가 + 부가세
```

집계는 `tenant_seat_usage`에 **high-water mark**(GREATEST로만 상승)로 쌓이고, 멤버 활성화·초대 활성화·청구서 조회·마감 시점에 현재 활성 수를 세어 갱신한다. 기록 월(`ym`)은 **그 이벤트가 일어난 서버 달력 월**이다.

> "무료"는 **기간이 아니라 좌석 수**다. "가입 첫 달 무료" 같은 시간 기반 무료체험은 코드에 없다 — 처음 5좌석이 늘 무료라는 뜻(활성 5명 이하면 매월 0원).

## 2. 케이스별 결제 상황 — [현행] 기준

가정: 인당 2,000원 / 무료 5좌석 / 6월(30일) 기준.

| # | 케이스 | [현행] 결과 |
|---|--------|-------------|
| C1 | 월초(6/1) 20명 활성 | 6월 = (20−5)×2,000 = **30,000** + VAT 3,000 = **33,000원** |
| C2 | **월말(6/30) 20명 활성** (질문 시나리오) | 하루만 활성이어도 6월 **전액 33,000원** — 일할 없음 |
| C3 | 6/1 10명 → 6/20 +10명(총 20) | 피크 20 기준 6월 **33,000원** (증원일 무관) |
| C4 | 6/1 20명 → 6/20 −8명(총 12) | 피크가 20으로 남음 → 6월 **33,000원** (감원해도 그 달은 안 내려감) |
| C5 | 무료→유료: 6/15 5명 → 6명 | 피크 6 → (6−5)×2,000 = 6월 **2,000** + VAT = **2,200원** |
| C6 | 유료→무료: 6/1 8명 → 6/25 4명 | 피크 8 → (8−5)×2,000 = 6월 **6,600원**, 7월부터 0원 |
| C7 | 월중 단가 변경(2,000→3,000, 미마감) | 잠정은 **조회 시점 단가**로 재계산 → 6월 잠정이 3,000 기준으로 바뀜 |
| C8 | 마감(확정) 후 인원·단가 변동 | 확정 청구서는 **불변**(스냅샷), 재마감 409 |
| C9 | 다음 달(7월) | 완전히 별도 달로 재계산. 6월 금액은 6월 청구서에만 귀속 |

**정리:** 현행은 "**그 달의 피크 좌석 × 월 단가**"라서 — 증원 타이밍·가입일이 금액에 영향을 안 주고(C1=C2=C3), 감원은 **당월에 반영 안 됨**(C4·C6).

---

## 3. 문제의식 — 왜 메리트가 약한가

- **월말 가입 = 한 달 전액**(C2): 6/30 가입 고객이 6월분을 통째로 내야 하니 **가입 유인이 떨어진다.**
- **증원 시점 무보상**(C3): 월 중간에 늘려도 그 달 전액이라, 고객은 "이왕이면 다음 달 1일에 늘리자"는 왜곡된 유인이 생긴다.
- 즉 현행은 **판매자에게만 유리하고 고객 체감이 나쁘다.** → 아래 일할계산 제안.

## 4. 개선 제안 — 일할계산(seat-day) 모델  ※[제안], 미구현

### 4-1. 핵심 아이디어
"그 달의 피크"가 아니라 **"매일의 과금좌석을 하루 단가로 누적"**한다. 그러면 가입일·증감 타이밍이 자연스럽게 금액에 반영된다.

```
그 달 일수 D              = 예) 6월 30, 7월 31, 2월 28/29
일 단가                   = 인당월단가 ÷ D            (2,000 ÷ 30 ≈ 66.7원)
그날 과금좌석(billable_d)  = max(0, 그날 활성좌석 − 무료좌석)
── 그 달 공급가 = round( (Σ_d billable_d) × 인당월단가 ÷ D )   ← 좌석-일 누적
   부가세       = round(공급가 × 10%)
   합계         = 공급가 + 부가세
```
> 누적 단위는 "**좌석-일(seat-day)**". 반올림은 일 단가에서 하지 않고 **월 합계에서 한 번**만 해 오차 누적을 막는다.

### 4-2. 전환의 **비대칭** 처리 (유료→무료는 이야기가 다르다)

증원(업그레이드)과 감원(다운그레이드)을 대칭으로 두면 다운그레이드 어뷰징(월초 잔뜩 → 월말 줄이기)이 생긴다. 그래서 상용 SaaS 관행을 따른다:

| 방향 | [제안] 처리 |
|------|-------------|
| **증원 / 무료→유료 (업그레이드)** | **전환일부터 일할 추가.** 늘린 그날부터 월말까지 좌석-일 누적 |
| **감원 / 유료→무료 (다운그레이드)** | **당월은 환불·감액 없음**, **다음 달 1일부터 반영.** 즉 그 달은 "그 달 도달한 최대 좌석 수준"을 유지 |

이 규칙이면 좌석-일 누적을 **"증가는 즉시, 감소는 이월"**로 계산한다(감소분은 당월 좌석 수를 낮추지 않고 다음 달부터 낮춘 값으로 시작).

### 4-3. 같은 케이스, [제안] 결과 (6월 30일 기준)

| # | 케이스 | [현행] | [제안] |
|---|--------|--------|--------|
| C1 | 6/1 20명 | 33,000 | 33,000 (한 달 내내 15좌석) |
| C2 | **6/30 20명** | 33,000 | 15좌석 × (1일/30) × 2,000 ≈ **1,000** + VAT ≈ **1,100원** |
| C3 | 6/1 10명 → 6/20 +10 | 33,000 | 5좌석×19일 + 15좌석×11일 = 좌석일 260 → 260×2,000/30 ≈ **17,333** + VAT ≈ **19,066원** |
| C4 | 6/1 20명 → 6/20 −8(총 12) | 33,000 | 감소는 당월 유지 → **33,000** (7월부터 7좌석 기준) |
| C5 | 6/15 5→6명 | 2,200 | 1좌석 × 16일 × 2,000/30 ≈ **1,067** + VAT ≈ **1,173원** |
| C6 | 6/1 8명 → 6/25 4명 | 6,600 | 감소 당월 유지 → 3좌석×30일 = **6,600원**, 7월 0원 |
| C9 | 7월 20명 유지 | 33,000 | 33,000 (전월 이월 없음) |

→ **월말 가입(C2)·월중 증원(C3·C5)에서 고객 체감이 크게 개선**되고, 다운그레이드 어뷰징(C4·C6)은 막힌다.

### 4-4. 도입 시 필요한 변경 (개략)
- 집계 테이블을 **"월 최대(max_seats)" → "좌석-일 누적(seat_days)"**로 확장하거나, **좌석 변동 이벤트 로그**(일자·증감)를 남겨 월말에 좌석-일을 산출.
- 증감 이벤트를 날짜와 함께 기록(현행은 "언제 몇 명"이 아니라 "피크"만 앎).
- 마감 스냅샷에 좌석-일·일수·전환 내역 포함.
- (선택) 자동 월마감 스케줄러 + 확정 청구서 알림/PG 청구.

## 5. 미결정 사항 (도입 전 정책 확정 필요)

1. **일할 기준일**: 전환 **당일 포함** 여부(6/30 전환 시 1일치 vs 0일치).
2. **다운그레이드 정책**: §4-2대로 "당월 유지 + 다음 달 반영"으로 확정할지, 아니면 **일할 환불**(대칭)로 갈지.
3. **무료 좌석 적용 단위**: 일할에서도 무료 5좌석을 **매일** 빼는지(제안), 월 단위로 빼는지.
4. **최소 과금/청구 주기**: 소액(수백 원) 발생 시 이월·합산 여부, 최소 청구액.
5. **테넌트 정지(SUSPENDED)/해지 시점**의 당월 처리.
6. **실제 결제 실행(PG)·결제일·알림** 도입 여부(현재 전부 미구현).
