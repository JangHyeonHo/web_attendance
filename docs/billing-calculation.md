# 결제(과금) 계산 방식 — 현행 동작 + 케이스별 정리 + 개선 제안

- 대상: `com.attendance.pro.billing` + `tenant_billing`/`tenant_seat_usage`/`invoice` (Flyway V22·V24)
- 이 문서의 구성
  1. **현행(As-Is)** 계산 방식 — 지금 코드가 실제로 하는 것
  2. **케이스별 결제 상황** — 가입·증원·감원·무료↔유료 전환·단가 변경 등에서 현행이 어떻게 동작하는가
  3. **문제의식** — 왜 현행(월 고정) 방식이 메리트가 약한가
  4. **확정 정책 v1** — "등록 시점 일할계산" (정책 확정, **✅ 구현 완료**)
  5. **남은 미결정 사항** — 구현 전 확인할 세부 정책
- ⚠️ 표기 규칙: **[현행]** = 지금 배포된 동작 / **[제안]** = 아직 구현 안 됨(정책 확정 후 별도 작업)

## 1. 현행(As-Is) 요약

| 항목 | 값 / 방식 |
|------|-----------|
| 과금 기준 | **활성 멤버 수(seat)** — `ACTIVE`·미삭제·`SYSTEM_ADMIN` 제외 |
| 집계 주기 | **달력 월**(`YYYY-MM`), 월중 **최대치(high-water mark)** |
| 인당 단가 | 기본 **2,000원/월** (VAT 별도) — 운영사가 회사별 조정 |
| 무료 좌석 | 기본 **5좌석** — 이 수까지 과금 제외 |
| 일할계산 | **없음** (그 달 하루라도 활성이면 피크 좌석으로 한 달 전액) |
| 확정(마감) | 운영사 **수동** 마감 시 스냅샷, 자동 스케줄러 없음 |
| 결제일·자동결제·알림 | **없음** — 청구서 계산·표시까지만 (PG 미연동) |

**[현행] 과금식** (`BillingService.compute`)
```
과금인원 = max(0, 월중최대활성 − 무료좌석)
공급가   = 과금인원 × 인당단가
부가세   = round(공급가 × 10%)
합계     = 공급가 + 부가세
```

집계는 `tenant_seat_usage`에 **high-water mark**(GREATEST로만 상승)로 쌓이고, 멤버 활성화·초대 활성화·청구서 조회·마감 시점에 현재 활성 수를 세어 갱신한다. 기록 월(`ym`)은 **그 이벤트가 일어난 서버 달력 월**이다.

> "무료"는 **기간이 아니라 좌석 수**다. "가입 첫 달 무료" 같은 시간 기반 무료체험은 코드에 없다 — 처음 5좌석이 늘 무료라는 뜻(활성 5명 이하면 매월 0원).

## 2. 케이스별 결제 상황 — [현행] 기준

가정: 인당 2,000원 / 무료 5좌석 / 6월(30일) 기준.

| # | 케이스 | [현행] 결과 |
|---|--------|-------------|
| C1 | 월초(6/1) 20명 활성 | 6월 = (20−5)×2,000 = **30,000** + VAT 3,000 = **33,000원** |
| C2 | **월말(6/30) 20명 활성** (질문 시나리오) | 하루만 활성이어도 6월 **전액 33,000원** — 일할 없음 |
| C3 | 6/1 10명 → 6/20 +10명(총 20) | 피크 20 기준 6월 **33,000원** (증원일 무관) |
| C4 | 6/1 20명 → 6/20 −8명(총 12) | 피크가 20으로 남음 → 6월 **33,000원** (감원해도 그 달은 안 내려감) |
| C5 | 무료→유료: 6/15 5명 → 6명 | 피크 6 → (6−5)×2,000 = 6월 **2,000** + VAT = **2,200원** |
| C6 | 유료→무료: 6/1 8명 → 6/25 4명 | 피크 8 → (8−5)×2,000 = 6월 **6,600원**, 7월부터 0원 |
| C7 | 월중 단가 변경(2,000→3,000, 미마감) | 잠정은 **조회 시점 단가**로 재계산 → 6월 잠정이 3,000 기준으로 바뀜 |
| C8 | 마감(확정) 후 인원·단가 변동 | 확정 청구서는 **불변**(스냅샷), 재마감 409 |
| C9 | 다음 달(7월) | 완전히 별도 달로 재계산. 6월 금액은 6월 청구서에만 귀속 |

**정리:** 현행은 "**그 달의 피크 좌석 × 월 단가**"라서 — 증원 타이밍·가입일이 금액에 영향을 안 주고(C1=C2=C3), 감원은 **당월에 반영 안 됨**(C4·C6).

## 3. 문제의식 — 왜 메리트가 약한가

- **월말 가입 = 한 달 전액**(C2): 6/30 가입 고객이 6월분을 통째로 내야 하니 **가입 유인이 떨어진다.**
- **증원 시점 무보상**(C3): 월 중간에 늘려도 그 달 전액이라, 고객은 "이왕이면 다음 달 1일에 늘리자"는 왜곡된 유인이 생긴다.
- 즉 현행은 **판매자에게만 유리하고 고객 체감이 나쁘다.** → 아래 일할계산 제안.

## 4. 확정 정책 v1 — "등록 시점 일할계산" (✅ 구현 완료)

> 정책 결정(2026-07): ① 일할은 **등록(증원) 시점 기준으로만**, ② 다운그레이드(감원·유료→무료)는 **당월 유지 + 다음 달 반영**, ③ 전환 **당일 제외**(익일부터 과금).
>
> **구현**(2026-07): `seat_change_event` 로그(V40) + `BillingService` 좌석일 재생 계산 + `invoice.seat_days/days_in_month` 스냅샷 + 청구서 화면 좌석-일 표기(V41). 단위 테스트 14건(이월 전액·월말등록 0원·월중 증원 일할·감원 당월 미반영·재증원).

### 4-1. 규칙

증감을 **비대칭**으로 처리한다.

| 방향 | 처리 |
|------|------|
| **증원 / 무료→유료 (업그레이드)** | 등록일 **다음 날부터** 월말까지 **일할 과금**(증가한 과금좌석만큼) |
| **감원 / 유료→무료 (다운그레이드)** | **당월은 감액·환불 없음**(그 달 도달한 좌석 수준 유지), 감소분은 **다음 달 1일부터** 반영 |

- 과금좌석(billable) = `max(0, 활성좌석 − 무료좌석)`. 무료 5좌석은 **인원 기준 공제**(어느 멤버가 무료인지 지정하지 않음).
- 증원 이벤트의 과금좌석 증가분 Δ = (증원 후 billable) − (증원 전 billable). 무료 구간(5명 이하)에서의 증원은 Δ=0.

### 4-2. 계산식 (좌석-일 누적)

```
D          = 그 달의 일수 (6월 30, 7월 31, 2월 28/29)
기초좌석    = 그 달 1일 시점의 과금좌석 (전월 말 상태에서 이월) — D일 전부 과금
증원분 Δ_i  = i번째 증원 이벤트의 과금좌석 증가 (등록일 e_i)
과금일수_i  = D − e_i        ← 등록 당일 제외(e_i+1 ~ 말일)
좌석일(seat-days) = 기초좌석 × D  +  Σ_i ( Δ_i × 과금일수_i )
공급가      = round( 좌석일 × 인당월단가 ÷ D )     ← 반올림은 월 합계에서 1회
부가세      = round(공급가 × 10%)
합계        = 공급가 + 부가세
다음 달 기초좌석 = max(0, 그 달 말 활성좌석 − 무료좌석)   ← 감원은 여기서 반영
```

### 4-3. 같은 케이스, 확정 정책 결과 (인당 2,000 / 무료 5 / 6월=30일)

| # | 케이스 | [현행] | [확정 v1] |
|---|--------|--------|-----------|
| C1 | 6/1 신규 20명 | 33,000 | 등록일(6/1) 제외 29일 → 15×29=435좌석일 → 29,000 + VAT = **31,900원** |
| C2 | **6/30 신규 20명** | 33,000 | 등록 익일=7/1 → 6월 과금일 0 → **6월 0원**, 7월 33,000 |
| C3 | 6/1 10명 → 6/20 +10(총 20) | 33,000 | (5×29) + (10×10) = 245좌석일 → 16,333 + VAT = **17,966원** |
| C4 | 6/1 20명 → 6/20 −8(총 12) | 33,000 | 감원 당월 무효 → 15×29=435 → **31,900원**, 7월 = 7×31 → **15,400원** |
| C5 | 6/15 5→6명 | 2,200 | 1좌석 × 15일 → 15좌석일 → 1,000 + VAT = **1,100원** |
| C6 | 6/1 8명(이월) → 6/25 4명 | 6,600 | 기초 3좌석 × 30 = 90 → **6,600원**, 7월 0원 |
| C9 | 7월 20명 유지(이월) | 33,000 | 기초 15 × 31 → **33,000원** (증감 없으면 전액) |

→ 월말 가입(C2)·월중 증원(C3·C5)에서 고객 체감이 개선되고, 다운그레이드 어뷰징(C4·C6)은 막힌다. 이월된(계속 유지되는) 좌석은 매달 전액(C9).

> ⚠️ **당일 제외**의 결과: 신규 등록월은 항상 최소 하루가 빠진다(6/1 등록도 29/30 = 31,900원, C1). 등록월 이후 계속 유지되는 좌석은 이월 기초좌석으로 **전액** 과금되므로 매달 하루씩 빠지지는 않는다. "등록월도 등록일 포함 전액"을 원하면 이 규칙만 조정하면 된다.

### 4-4. 구현 시 필요한 변경 (개략)
- **좌석 변동 이벤트 로그** 신설 — 일자 + 증감(+/−) 기록. 현행 `tenant_seat_usage`는 "피크"만 알아 등록일별 일할을 못 낸다.
- 월 청구 계산을 **기초좌석 × D + Σ(증원 Δ × 과금일수)** 누적으로 교체(`BillingService.compute`).
- 감원은 당월 미반영·다음 달 기초좌석에만 반영하는 이월 로직.
- 마감 스냅샷에 좌석-일·일수 포함(`invoice` 확장).
- 프론트 청구서 표기(좌석-일/일할 내역) 및 3개 언어 라벨.

## 5. 남은 미결정 사항 (구현 전 확인)

정책 3종은 확정(§4). 아래는 세부 확인 필요:

1. **무료 좌석 공제 방식**: 과금좌석을 매 이벤트 시점의 `활성−무료`로 산정(현재 문서 가정). 특정 멤버를 무료로 고정하는 방식은 배제.
2. **등록월 당일 포함 여부 재확인**: §4-3 ⚠️ 참고 — 6/1 등록도 29일 과금이 되는데 이대로 둘지.
3. **최소 청구액**: 소액(수백 원) 발생 시 이월·합산·최소액 정책.
4. **테넌트 정지(SUSPENDED)/해지 시점**의 당월 처리.
5. **실제 결제 실행(PG)·결제일·알림**: 현재 전부 미구현 — 이번 범위에 포함할지.
