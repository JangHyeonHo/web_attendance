## D21. 멤버 온보딩 — 초기 비밀번호 1회 표시 → 이메일 초대(본인 확인) 전환

- **배경**(소유자 요구): 관리자가 이메일로 멤버 추가 → 본인 확인 메일 → 링크에서 비밀번호 설정 → 사용 가능. 오송신 방지 확인 체크와 오송신 수습(정지/삭제) 필요. 비밀번호 찾기도 같은 메커니즘(본문 상이), 변경 후 재로그인 필수.
- **핵심 결정**:
  - **토큰**: 랜덤 48B → **SHA-256 해시만 DB 저장**(user_token), 1회용, TTL 초대 72h/재설정 30m. 재발송·정지·삭제 시 즉시 무효화. 응답·로그에 원문 금지.
  - **상태 전이**: 등록 시 `PENDING`(Phase 1에서 예약해둔 값) → 비밀번호 설정 완료 시 `ACTIVE`. PENDING은 로그인 불가(기존 authenticate가 ACTIVE만 허용 — 코드 무변경으로 성립).
  - **오송신 방지 UX**: 폼 제출 = 발송이 아니라 **입력 이메일 재확인 인라인 패널**을 먼저 연다. 수습은 소프트 삭제(`deleted` 플래그 — 출결 기록 보존, 삭제 시 토큰 무효화·세션은 기존 재검증이 즉시 회수). 소프트 삭제 행이 `UNIQUE(tenant_id,email)`을 점유해 재등록을 막는 함정은 **email_key 생성 컬럼**으로 해소(교차 검증 CR3 발견).
  - **재로그인 강제**: `users.password_changed_at(DATETIME(6))` + 세션 발급 시각 비교를 기존 `SessionRevalidationInterceptor`에 추가. 초 단위 정밀도로는 같은 초 발급 세션이 비껴가는 레이스가 실측되어 **마이크로초(6) 정밀도** 채택.
  - **비밀번호 찾기의 테넌트 스코프**: 같은 이메일이 복수 테넌트에 존재 가능 → 요청에 테넌트 식별 필수(서브도메인이면 호스트 자동/루트는 코드 입력). 계정 존재 비노출(성공 응답 통일 202) + 전용 레이트 리밋.
  - **URL 없는 SPA와 메일 링크의 화해**: 링크는 `/?token=...` 단일 쿼리 파라미터 — 프론트가 기동 시 읽고 `history.replaceState`로 즉시 제거 후 W010 전개. URL 라우팅 도입이 아님(서버 주도 화면 전개 유지).
- **기대값**: 비밀번호가 관리자 화면·메일 어디에도 평문 노출되지 않음(본인만 설정). 오송신은 발송 전 1회 확인 + 발송 후 삭제로 이중 수습.
- **리스크/비용**: 메일 인프라 의존(발송 실패 시 멤버는 생성되고 `mailSent=false` + 재발송 버튼 — 등록 트랜잭션과 분리). 초대 만료(72h) 후 재발송 운영 필요.

## D22. 메일 발송·템플릿 — SMTP 환경변수 + 템플릿 DB(관리 화면 미리보기)

- **결정**(소유자 확정): 템플릿은 코드가 아닌 **DB(mail_template)** — 용도(INVITE/RESET)×언어(KOR/ENG/JPN) 6행 Flyway 시드, **SYSTEM_ADMIN 관리 화면(W012)에서 수정·샘플 값 미리보기**. 언어 마스터와 같은 "DB 단일 출처 + 화면에서 수정 즉시 반영" 패턴의 재사용.
- **메일 언어 = 테넌트 소재국**(KR→한국어, JP→일본어, 그 외 영어 폴백) — 수신자 언어를 모르는 시점이므로 회사 언어가 기준(소유자 확정).
- 변수 치환(`{memberName}` `{tenantName}` `{actionUrl}` `{expiresAt}` `{inviterName}`)은 저장·미리보기 시 2층 검증(미지 변수·필수 변수 누락).
- **발송 구현**: 운영 = SMTP(spring-boot-starter-mail, `SMTP_HOST/PORT/USER/PASS/FROM` + `APP_MAIL_LINK_BASE_URL` — prod 미설정 시 기동 실패). 개발 = **LoggingMailSender**(메일 전문을 로그로 — 로컬 검증·E2E가 로그에서 링크를 취득, 소유자의 테스트용 Resend와 병행 가능).
- **리스크**: 템플릿을 화면에서 수정 가능 = 오타·변수 삭제 사고 가능 → 미리보기·치환 검증이 방어. 메일 제목의 개행 주입(헤더 인젝션)은 발송 계층에서 차단.

## D23. 근무 스케줄·법정 휴게·공휴일

- **개인 기본 스케줄**: `users.default_work_start/end`(기본 09:00/18:00). 우선순위 **work_schedule(일자 오버라이드) > 개인 기본 > 상수**. 멤버 초대 시 입력, 이후 `PUT /members/{id}/schedule`.
- **법정 휴게**(소유자 요구 — 국가별 법정 기준): 국가는 tenant.country(V7에서 tenant로 승격 — profile은 미등록 가능이라 부적합).
  - KR 근로기준법 §54: 4시간 → 30분, 8시간 → 1시간 (도달 시 포함)
  - JP 労働基準法 §34: 6시간 **초과** → 45분, 8시간 **초과** → 1시간
  - **총 근무시간 = 체류(출근~퇴근) − max(법정 휴게, 실제 휴식 합)** — 실제 휴식이 법정을 초과하면 초과분만큼 차감(소유자 명시 규칙), 휴식 미기록이어도 법정 휴게는 차감(법이 보장하는 시간).
  - BREAK 스탬프 페어링: 시작→종료 짝, 미종료 휴식은 퇴근까지로 간주(근무시간 부풀림 방지).
- **공휴일**: `holiday`에 이름·유형(NATIONAL/COMPANY) 추가. **국가 공휴일은 Nager.Date 공개 API 동기화**(소유자 승인 — 테넌트 생성 시 당해·익년 자동, 실패 허용 + W013 수동 동기화. 빈 응답이 기존 데이터를 지우지 못하게 검증 후 원자 교체, COMPANY 행 불가침). **회사 지정 공휴일**(창립기념일 등)은 TA가 W013에서 CRUD. 월별 상세에 공휴일 이름 표시.
- **리스크**: 외부 API 의존 — 이 개발 환경에서는 프록시 정책상 실호출이 차단되어(제어된 502 확인) **운영망에서 1회 실검증 필요**. 법정 휴게는 "스케줄 기준 산출"이라 실근무가 스케줄과 크게 다른 날의 엄밀 산출은 후속 과제(문서 명기).
